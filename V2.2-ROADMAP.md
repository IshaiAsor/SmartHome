## Appendix A — Approved queue design (binds F1.4, F6, F8)

Full plan: `C:\Users\ishai\.claude\plans\yes-keen-petal.md` (+ flow sketch `yes-keen-petal-sketch.txt`). Approved 2026-06-05. Key decisions, so this file is self-contained:

**1. Static routing keys — no userId anywhere in RKs** (userId lives in the payload only):

```
telemetry.arrived · rules.evaluate · pipeline.trigger · pipeline.result ·
device.state.changed · action.dispatch
pipeline.stage.sensor_digest · pipeline.stage.command_exec · pipeline.stage.done.v1
pipeline.stage.{kind}.{name}.{version}        ← only dynamic part: ML model identity
```
All queue bindings are exact match (no `.*` wildcards). Shared `DLQ_ARGS` export (dead-letter exchange + 5-min TTL) applied to **every** queue assertion, static and dynamic.

**2. Async queue-based pipeline stages** (replaces the snapshot's synchronous HTTP for-loop to ml-router — that version must NOT be re-landed):
- Pipeline-worker has **4 consumers**: trigger (creates PipelineRun, publishes first stage), sensor_digest, command_exec, finalization (`q.pipeline.stage.done` → update run, publish `pipeline.result`, socket emit).
- **ml-router is a queue consumer, not an HTTP service**: `setupModelQueues()` asserts one queue per registered ML model at startup (`q.pipeline.stage.{kind}.{name}.{version}`), prefetch 1 for LLM / 4 for VLM.
- `advancePipeline()` helper (~45 lines, inlined in both pipeline-worker and ml-router): writes PipelineRunStage, then publishes next stage's queue or `pipeline.stage.done` (status completed/failed). Errors → done with `failed` → message lands in DLQ.
- Why: LLM inference can take 60+ s — HTTP blocks, no retry; per-stage queues give independent backpressure, DLQ, and per-model scaling.

**3. SensorBlueprint** (new Prisma model): `sensor_groups` JSON (named capability groups), `profile` JSON (target ranges), optional `camera_device_id`; `Pipeline.blueprint_id` FK.
- Validation **hard at pipeline creation**, trusted at runtime: any vlm/llm stage → blueprint required; any vlm stage → blueprint must have `camera_device_id`.
- `sensor_digest` stage reads groupings/profile from the blueprint (not from flat stage config); VLM/LLM handlers read the cached camera frame via the blueprint's camera device.
- Stage chains: `vlm → sensor_digest → llm → command_exec` (with camera) or `sensor_digest → llm → command_exec` (sensors only).

⚠️ The plan's JSON examples use farming terms (`plant_type`, `nutrients`…) — predates the generic-naming rule. When implementing, all examples/seeds must be domain-agnostic.

**Where it lands in the roadmap:** F1.4 builds `packages/queue` with the static RKs/DLQ_ARGS/payload types from day one (no migration later) · F6 workers consume them · F8.1 adds SensorBlueprint + validation · F8.2 the 4 consumers + advancePipeline · F8.3 ml-router model queues.
